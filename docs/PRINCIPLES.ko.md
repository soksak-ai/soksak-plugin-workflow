# 개발 원칙 (한글 번역본 — 정본은 PRINCIPLES.md)

이 저장소의 모든 구현과 변경은 이 규칙에 구속됩니다. 규칙이 먼저고 코드가 따릅니다. 코드와
규칙이 어긋나면 코드를 고칩니다. 규칙 자체가 틀렸으면 조용히 완화하지 말고 이 문서를 명시적
커밋으로 정정합니다.

Grounding: Rust 엔진은 사이드카 정본 repo `soksak-sidecar-workflow` 에 있고, 이 repo 는 이제 순수
JS 다. 여기 나오는 `src/…`·`workflows/…`·`references/…`·`e2e/…`·`tools/…` 경로는 그 사이드카
repo 를 가리킨다.

## 1. directive 단일 진실
저작 게이트를 통과한 정련본이 정본입니다. 사람이 보는 표면(보드의 덩어리 설명)과 기계가 쓰는
검증 기준(exec 주입 directive)은 같은 문자열이어야 합니다. 유일한 예외는 사용자가 명시로 준
directive 입니다.

## 2. fail-loud
게이트 실패는 발행·완료 거부입니다. 침묵 폴백과 빈 성공을 금지합니다. 방어 파서는 유지하되
계약 위반은 반드시 관찰 가능해야 하며, 검증 층(저작·--emit·exec-stage·relay)은 어느 것도
제거하지 않습니다.

## 3. 정규화 3수준
전역 템플릿은 콘텐츠 주소(1행, byte 안정), 청크 공유값(directive)은 참조, 노드에는 작은 고유
vars 만 둡니다. 공유 프롬프트의 byte 안정성이 dedup 의 전제입니다 — 다듬기 변경은 의도적일
때만 합니다.

## 4. 두 축 분리
검증 축은 badge(검수전 → o/x/f), 완료 축은 status 입니다. badge 를 가진 노드의 완료 판정은
badge 뿐입니다. 새 검증 대상 kind 는 badge 축을 재사용합니다.

## 5. 이슈라이즈 게이트
드래프트는 이슈 분해가 아닙니다. 파일별 실코드화 task 는 오직 issuerize 를 통해서만 생기며,
게이트는 badge='o' ∧ fact 전부 확정 ∧ plan-unit 전부 확정입니다. 우회 경로를 만들지 않습니다.

## 6. 기준 약화 금지
기준 미달이면 구현과 픽스처를 고칩니다 — 기준을 낮추지 않습니다. 단언 완화·skip 추가·임계
완화는 전부 위반입니다. 기준 자체가 틀렸으면 이 문서와 함께 명시적 커밋으로 정정합니다.

## 7. 불필요한 LLM 단계 금지
LLM 은 정련·발굴·판정에만 씁니다. 이미 정련된 입력을 재정련하지 않습니다. 결정적으로 계산
가능한 것(원장 조립·id 매핑·게이트 판정)은 코드가 합니다.

## 8. 지시어가 본체
결과 품질은 주입 기계가 아니라 지시어에서 나옵니다 — 그 기계가 server2 의 사인이었습니다.
스텝·방법론 추가는 canonical doc 과 지시어 원문으로 합니다. 실행기 확장은 실측이 필요를
증명할 때만 합니다.

## 9. 메아리 금지
산출 스키마에 부모 데이터 사본을 두지 않습니다. 부모 정보는 실행기가 프롬프트로 조성하고,
산출은 그 스텝의 자기결정만 담습니다. 사본이 없으면 drift 는 없습니다.

## 10. 사후분해
default 는 통합 한턴입니다. 분해는 통합 한턴의 결정성 실패가 실측으로 증명될 때만 채택합니다.
분해 위상(병렬/체인/입력 선별)은 코드가 아니라 방법론 doc(데이터)이 정합니다.

## 11. 생성은 지시어, 검증은 기계
계약은 지시어 문장과 사례로 가르칩니다. 위반은 결정적 사후 대조(원장 id 대조·중복/미배정
검출)가 거부합니다. 스키마는 구조만 강제합니다.

## 12. ground 는 o 확정만

스테이지가 그 위에 **쌓아 올리는** 원장·{{facts}} ground(design 체인·plan·body)에는 o 확정만
싣는다. f(치명)·x(반려)는 ground 가 아니다 — 섞어 주입하면 모순 기반을 쥐여주는 것(실측:
혼합 111건을 받은 plan 은 쓰레기 1건, o 만 받은 같은 턴은 실질 파일 유닛 39건). 자기 임무에
전체 원장이 필요한 스테이지는 유지한다: hunt(중복 회피)·classify(전수 배정)·audit(f 집계).

## 13. 모든 층의 이원 인증

노드별 검증(부품)은 집합 인증(조립)을 대신하지 못한다. draft 층=verify+audit, plan 층=
유닛 검증+plan-audit(파일 집합이 빌드·배포·**운영** 가능한 시스템으로 조립되는가 — auth·
관리 콘솔·조회 화면·배치·부트 배선). 형식 커버리지 100%(전 요건이 어느 유닛에 인용)여도
조립체는 미완일 수 있다(실측: 60/60 인용인데 콘솔·auth 표면 부재). 조립 검사 없는 층은
미완의 층이다.

## 14. 부결은 보정 루프를 태운다

구체 gaps 를 동반한 "미완" 판정은 폐기가 아니다 — gaps 를 기계적으로(저작 LLM 0) 후보로
재투입하고, 기존 검증 파이프가 분별하며(가짜 gap 은 x 로 소멸), 인증을 재실행한다. 라운드
상한(2), 소진 시 fail-loud 폐기. 소비자 없는 gaps 필드는 결함이다.

## 15. 재시도는 transient 에만

재시도 루프가 삼켜도 되는 것은 529/overloaded/timeout 뿐 — 결정적 실패(파싱 에러·미해석
플레이스홀더·계약 위반)는 절대 아니다. 결정적 실패의 재시도는 결함을 일시 장애로 위장하고
예산을 태운다(실측: 템플릿 결함에 8라운드 소진). 에러를 먼저 분류하고, 결정적 부류는
즉시 크게 실패하라.

## 16. 범위는 언급이 아니라 필수다

지시문이 이름을 붙이지 않은 표면이라도, 그것 없이는 결과물이 실제로 운영될 수 없거나 책임을
물을 수 없다면 범위 안이다 — 판정은 요청의 문구가 아니라 make-or-break 테스트가 내린다. 그
사정거리는 결과물이 함의하는 모든 액터에 미친다: 결과물을 사용하는 무대 위 액터와, 결과물을
운영하는 무대 뒤 액터(도메인에 따라 플랫폼 운영자, 라이브옵스, 배포 주체, 관리자 — 만들어지는
모든 것은 자기 운영 층을 함의한다). 범위 밖에 남는 것은 "있으면 좋은 것"뿐이다. 실측: 전 파이프
완주가 명시 요건 60/60을 덮고도 운영자 층을 전혀 내지 않았다 — 지시어가 발굴을 언급 기준으로
묶어 두었기 때문이다(2026-07-10).
