# 개발 원칙 (한글 번역본 — 정본은 PRINCIPLES.md)

이 저장소의 모든 구현과 변경은 이 규칙에 구속됩니다. 규칙이 먼저고 코드가 따릅니다. 코드와
규칙이 어긋나면 코드를 고칩니다. 규칙 자체가 틀렸으면 조용히 완화하지 말고 이 문서를 명시적
커밋으로 정정합니다.

## 1. directive 단일 진실
저작 게이트를 통과한 정련본이 정본입니다. 사람이 보는 표면(칸반 덩어리 설명)과 기계가 쓰는
검증 기준(exec 주입 directive)은 같은 문자열이어야 합니다. 유일한 예외는 사용자가 명시로 준
directive 입니다.

## 2. fail-loud
게이트 실패는 발행·완료 거부입니다. 침묵 폴백과 빈 성공을 금지합니다. 방어 파서는 유지하되
계약 위반은 반드시 관찰 가능해야 하며, 검증 층(저작·--emit·exec-stage·relay)은 어느 것도
제거하지 않습니다.

## 3. 정규화 3수준
전역 템플릿은 콘텐츠 주소(1행, byte 안정), 청크 공유값(directive)은 참조, 노드에는 작은 고유
vars 만 둡니다. 공유 프롬프트의 byte 안정성이 dedup 의 전제입니다 — 다듬기 변경은 의도적일
때만 합니다.

## 4. 두 축 분리
검증 축은 badge(검수전 → o/x/f), 완료 축은 status 입니다. badge 를 가진 노드의 완료 판정은
badge 뿐입니다. 새 검증 대상 kind 는 badge 축을 재사용합니다.

## 5. 이슈라이즈 게이트
드래프트는 이슈 분해가 아닙니다. 파일별 실코드화 task 는 오직 issuerize 를 통해서만 생기며,
게이트는 badge='o' ∧ fact 전부 확정 ∧ plan-unit 전부 확정입니다. 우회 경로를 만들지 않습니다.

## 6. 기준 약화 금지
기준 미달이면 구현과 픽스처를 고칩니다 — 기준을 낮추지 않습니다. 단언 완화·skip 추가·임계
완화는 전부 위반입니다. 기준 자체가 틀렸으면 이 문서와 함께 명시적 커밋으로 정정합니다.

## 7. 불필요한 LLM 단계 금지
LLM 은 정련·발굴·판정에만 씁니다. 이미 정련된 입력을 재정련하지 않습니다. 결정적으로 계산
가능한 것(원장 조립·id 매핑·게이트 판정)은 코드가 합니다.

## 8. 지시어가 본체
결과 품질은 주입 기계가 아니라 지시어에서 나옵니다 — 그 기계가 server2 의 사인이었습니다.
스텝·방법론 추가는 canonical doc 과 지시어 원문으로 합니다. 실행기 확장은 실측이 필요를
증명할 때만 합니다.

## 9. 메아리 금지
산출 스키마에 부모 데이터 사본을 두지 않습니다. 부모 정보는 실행기가 프롬프트로 조성하고,
산출은 그 스텝의 자기결정만 담습니다. 사본이 없으면 drift 는 없습니다.

## 10. 사후분해
default 는 통합 한턴입니다. 분해는 통합 한턴의 결정성 실패가 실측으로 증명될 때만 채택합니다.
분해 위상(병렬/체인/입력 선별)은 코드가 아니라 방법론 doc(데이터)이 정합니다.

## 11. 생성은 지시어, 검증은 기계
계약은 지시어 문장과 사례로 가르칩니다. 위반은 결정적 사후 대조(원장 id 대조·중복/미배정
검출)가 거부합니다. 스키마는 구조만 강제합니다.
